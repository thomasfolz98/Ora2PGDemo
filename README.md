# OracleDemo – Oracle 21c XE → PostgreSQL 15 Migration

Containerisierte Demo einer Oracle-nach-PostgreSQL-Migration mit Ora2Pg.
Zielgruppe: Unternehmen, die Oracle-Lizenzkosten reduzieren oder Workloads
auf PostgreSQL konsolidieren möchten.

Zwei Demo-Stufen zeigen den Einstieg und die volle Komplexität einer realen Migration:

| Demo     | Schema           | Komplexität | Patches | API-Port |
| -------- | ---------------- | ----------- | ------- | -------- |
| Basic    | `app_basic_demo` | 2 Tabellen, 1 Index, 1 Trigger | keine | 8081 |
| Advanced | `app_demo`       | 3 Tabellen, View, 2 Trigger, PL/SQL-Package | 1 | 8082 |

## Architektur

| Rolle              | Image / Build                     | Arch        | Hinweis                                 |
| ------------------ | --------------------------------- | ----------- | --------------------------------------- |
| Quell-DB (Oracle)  | `gvenzl/oracle-xe:21-slim`        | linux/amd64 | Rosetta-Emulation auf Apple Silicon     |
| Ziel-DB (Postgres) | `postgres:15`                     | arm64/amd64 | Nativ                                   |
| Ora2Pg Basic       | Custom (Ubuntu 22.04, Ora2Pg v24) | linux/amd64 | Oracle Instant Client 21.12             |
| Ora2Pg Advanced    | Custom (Ubuntu 22.04, Ora2Pg v24) | linux/amd64 | Oracle Instant Client 21.12             |
| API Basic          | `python:3.12-slim` + FastAPI      | arm64/amd64 | Port 8081, schema app_basic_demo        |
| API Advanced       | `python:3.12-slim` + FastAPI      | arm64/amd64 | Port 8082, schema app_demo + pkg_faktura |

Beide Schemas werden beim ersten Oracle-Start automatisch aus den Scripts
unter `docker/oracle/init/` angelegt (nummeriert `10_`–`12_` für Basic,
`20_`–`25_` für Advanced).

## Plattform-Kompatibilität

Oracle XE 21c existiert nur als `linux/amd64`-Image. Das beeinflusst je nach Host-Plattform den Betrieb:

| Plattform | Status | Hinweis |
| --------- | ------ | ------- |
| **macOS Apple Silicon** (M1/M2/M3) | ✅ getestet | Oracle + Ora2Pg laufen via Rosetta-Emulation (`platform: linux/amd64`) |
| **macOS Intel** | ✅ erwartet | x86_64-nativ, kein Rosetta nötig |
| **Ubuntu / Debian x86_64** | ✅ erwartet | Läuft nativ, kein Änderungsbedarf an `docker-compose.yml` |
| **Windows x86_64** | ⚠️ möglich | Docker Desktop + WSL2 erforderlich; Bash-Scripts im WSL2-Terminal ausführen |
| **Windows ARM** | ⚠️ ungetestet | Emulation wie Apple Silicon; möglicherweise Performance-Einbußen |

> Auf Linux x86_64 kann das `platform: linux/amd64`-Flag in `docker-compose.yml` entfernt werden — es ist dort redundant, schadet aber nicht.

## Voraussetzungen

- Docker Desktop (oder Docker Engine auf Linux)
- ~4 GB freier RAM für den Oracle-Container
- Bash-Shell (macOS/Linux nativ; Windows: WSL2-Terminal)

## Quickstart

```bash
# Einmalig: Images bauen und alle Container starten
docker compose up -d --build

# Oracle braucht beim Erststart ~3 Minuten
# Status prüfen:
docker ps
```

### Demo 1 – Basic (saubere Migration, kein Patch)

```bash
./basic/migrate.sh
```

Swagger-UI: http://localhost:8081/docs

Was passiert:
- Ora2Pg exportiert DDL (TABLE + TRIGGER) ohne Warnungen
- Kein manuell geschriebenes SQL notwendig
- 5 Produkte + 8 Bestellungen landen sauber in PostgreSQL

Rollback: `./basic/rollback.sh`

### Demo 2 – Advanced (realistische Komplexität)

```bash
./advanced/migrate.sh
```

Swagger-UI: http://localhost:8082/docs

Was passiert:
- Ora2Pg-Cost Level B-5, ~1 PT Aufwand
- PL/SQL-Package `PKG_FAKTURA` wird zu PL/pgSQL konvertiert
- 1 manueller Patch nötig: `NUMBER`-Rückgabetyp → `numeric`
- 10 Kunden, 15 Rechnungen, 20 Positionen
- Package-Funktionen direkt über API aufrufbar (`/kunden/{id}/umsatz`)

Rollback: `./advanced/rollback.sh`

## Typ-Mapping (automatisch via Ora2Pg)

| Oracle        | PostgreSQL                        |
| ------------- | --------------------------------- |
| `VARCHAR2(n)` | `varchar(n)`                      |
| `NUMBER(p,s)` | `decimal(p,s)`                    |
| `NUMBER(p)`   | `bigint` (p≤18) / `numeric(p)`    |
| `DATE`        | `timestamp(0)`                    |
| `CLOB`        | `text`                            |
| `SYSDATE`     | `statement_timestamp()`           |
| `IDENTITY`    | `bigint GENERATED BY DEFAULT AS IDENTITY` |
| `NVL(a,b)`    | `COALESCE(a,b)`                   |

## Bekannte Stolperfallen

- `docker exec` braucht `-i`, sobald stdin reingereicht wird (heredoc/Redirect).
- Nach `COPY` immer die Identity-Sequenzen mit `setval()` nachziehen.
- Ora2Pg: `CONCAT`-Warnung in Views ist bei `||`-Verkettung ein False Positive.
- Oracle XE 21c gibt es nur als x86_64 → Rosetta auf Apple Silicon ist Pflicht.
- Port 8000 ggf. schon belegt (z.B. durch Portainer) → API auf 8082/8081 gemappt.

## Reset (Neustart von Null)

```bash
docker compose down
rm -rf docker/oracle/data/* docker/postgres/data/*
docker compose up -d
# Oracle neu-initialisiert sich aus docker/oracle/init/
```
